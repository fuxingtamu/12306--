            .flatMap(train -> buildResults(train, condition).stream())

## 4. AI输出 / 工具输出示例

### 修复代码（diff格式）

```diff
 public List<TrainVO> queryTrains(TrainQueryDTO condition) {
     validateCondition(condition);
     List<Train> trains = findTrainsByRoute(condition);

     List<TrainVO> results = trains.stream()
             .flatMap(train -> buildResults(train, condition).stream())
-            .filter(result -> condition.isOnlyAvailable()
-                    || result.getAvailableCount() > 0)
+            .filter(result -> !condition.isOnlyAvailable()
+                    || (result.getAvailableCount() != null
+                        && result.getAvailableCount() > 0))
             .toList();

     results.sort(Comparator.comparing(TrainVO::getStartTime));
     return results;
 }
```

### 修复说明

| 变更 | 说明 |
|------|------|
| `condition.isOnlyAvailable()` → `!condition.isOnlyAvailable()` | 取反：当不勾选"只看有票"时，表达式为true，保留全部 |
| `result.getAvailableCount() > 0` → `(result.getAvailableCount() != null && result.getAvailableCount() > 0)` | 增加null防御：若数据异常导致availableCount为null，视为无票过滤 |

**布尔逻辑解读**：

```
!onlyAvailable = true  →  保留全部（不勾选筛选时）
!onlyAvailable = false →  需判断 availableCount：
    null      →  false（过滤）
    0         →  false（过滤）
    > 0       →  true（保留）
```

### 影响范围分析

| 影响对象 | 影响分析 | 风险 |
|----------|----------|------|
| `TicketQueryServiceImpl.filterAvailable()` | 直接修改 | 低——仅改布尔表达式 |
| `filterAvailable` 的调用方 `queryTrains()` | 调用不变 | 无 |
| 前端 `TicketQuery.vue` 的"只看有票"复选框 | 前端传 `onlyAvailable: true/false`，修复后行为正确 | 无——前端无感知 |
| 其他可能的筛选方法 | 若 `OrderServiceImpl` 等有类似筛选逻辑，一并排查 | 中——需全局搜索 `/|/|.*Available` 模式 |
| API响应 JSON | 过滤后返回的车次数量减少（onlyAvailable=true时） | 低——符合用户预期 |

### 回归测试点

| 编号 | 测试场景 | 输入 | 预期 |
|------|----------|------|------|
| RT-01 | 勾选有票+有票车次 | `onlyAvailable=true`, `count=5` | 保留 |
| RT-02 | 勾选有票+无票车次 | `onlyAvailable=true`, `count=0` | **过滤**（缺陷验证） |
| RT-03 | 不勾选+有票车次 | `onlyAvailable=false`, `count=5` | 保留 |
| RT-04 | 不勾选+无票车次 | `onlyAvailable=false`, `count=0` | 保留 |
| RT-05 | 勾选有票+count=null | `onlyAvailable=true`, `count=null` | **过滤**（null防御） |
| RT-06 | 不勾选+count=null | `onlyAvailable=false`, `count=null` | 保留 |
| RT-07 | 混合列表 | [G101(0票), G102(5票), G103(3票)] + onlyAvailable=true | 保留G102和G103 |
| RT-08 | 全部无票 | [G101(0票), G102(0票)] + onlyAvailable=true | 返回空列表 |

## 5. 使用建议

1. **修复后不要只跑1个测试**：案例11的4个测试用例 + 本案例的8个回归测试全部通过才算修复完成
2. **null防御是AI常漏的点**：要求在过滤条件中显式处理null，避免NPE
3. **全局搜索同类缺陷**：`grep -r "isOnlyAvailable.*||/|isOnly.*||"` 搜索是否有其他类似的布尔表达式错误
4. **最小修改原则**：只改 `!` 和 null 检查两处，不动 `flatMap`、`sort` 等无关代码
5. **记录缺陷根因**：在GitHub Issue中注明"`||` 误用作 `&&` 语义"，帮助后续Code Review关注同类模式
