# 案例01：AI辅助设计"车票查询"用例实现流程

> **适用章节**：10.4.2 设计用例实现方案
> **工具形式**：LLM Chat（Claude Opus 4.8 / GPT-4o，VS Code）+ 结构化提示词 + 人工多轮审阅迭代
> **工作范式**：🧑 人机多轮交互 —— 人主导设计方向，AI快速产出结构化初稿，人审阅精化
> **案例功能**：车票查询
> **来源文档**：12306_AI辅助软件开发案例_10-13章_全量补齐版

---

## 1. 案例背景

车票查询是12306软件中的高频核心功能。用户输入出发地、目的地和出发日期后，系统查询符合条件的车次，展示出发/到达时间、历时、席别、余票和票价。

本案例对应项目中 [TrainController.query()](../../backend/src/main/java/com/railway/controller/TrainController.java) 和 [TicketQuery.vue](../../frontend/src/views/TicketQuery.vue) 的实际实现。

---

## 2. 设计思想

在详细设计阶段，将需求转换为可执行的用例实现方案。AI快速形成结构化设计草稿，梳理主事件流、备选事件流、参与类和职责划分。提示词明确业务背景、设计原则和输出格式，避免大模型泛泛而谈。

**提示词设计四要素**：
1. 明确阐述任务需求——设计"车票查询"用例实现
2. 简洁介绍设计原则——区分边界/控制/实体类，从用例流程出发
3. 提供具体的软件上下文——出发/到达站名、席别枚举、余票实时查询
4. 提供输出约束——用例名称/参与者/前置条件/主事件流/备选事件流/参与类

## 3. 工具与方法选型

### 3.1 为什么选择 LLM Chat + 结构化提示词？

本案例是**设计阶段**任务——将模糊的需求转化为结构化的用例实现方案。这类任务的本质是"人在做设计决策，AI在做文档起草"。选型分析如下：

| 对比维度 | LLM Chat + 提示词（✅ 本案例选择） | Agent 自主执行（❌ 不适合） |
|----------|-----------------------------------|---------------------------|
| **设计决策权** | 人主导：提示词里定原则、审输出、做取舍 | Agent 自主规划，人只在最后审查 |
| **领域知识注入** | 人在提示词中注入预售期、车站数据等业务知识 | Agent 需自行探索项目，可能遗漏隐性业务规则 |
| **输出可控性** | 结构化提示词约束输出格式，偏差可控 | Agent 可能"跑偏"——生成过度设计的方案 |
| **迭代效率** | 人看输出→发现问题→调整提示词→重新生成，每轮5-10分钟 | Agent 完成全流程后人审查，返工代价大 |
| **适用场景** | 创意性、判断性、需要人拍板的阶段 | 机械性、规则明确、可自动验证的阶段 |

**核心判断**：用例实现方案涉及"哪些类参与协作"、"备选流覆盖哪些异常"、"职责边界画在哪里"——这些都是需要人的经验和判断力的决策。LLM Chat 模式下，人每次对话都是一次"微决策"，而不是把决策权交给 Agent。**设计阶段，人必须是驾驶员，AI是副驾。**

### 3.2 为什么不用 IDE 内联补全（Copilot）？

Copilot 适合补全**局部代码块**（如案例08的脱敏方法），但本案例需要的是**跨多个类的全局设计文档**。Copilot 没有"结构化提示词"的机制，无法通过一段 prompt 约束输出格式为"用例名称→参与者→前置条件→事件流→参与类"。LLM Chat 的**长上下文 + 格式化输出**能力是刚需。

### 3.3 具体工具链

| 组件 | 本案例使用 | 说明 |
|------|-----------|------|
| **模型** | Claude Opus 4.8（推荐）或 GPT-4o | 需要强推理能力来梳理事件流和类职责；Opus 4.8 在结构化输出和遵循设计原则方面表现更好 |
| **IDE** | VS Code | 配合 Claude Code 扩展或 Web Chat，方便对照项目代码验证 AI 输出的类名 |
| **辅助工具** | draw.io | 顺序图需手工绘制（或用 Mermaid），本案例中顺序图是在 AI 输出基础上人工调整的 |
| **输入方式** | 结构化提示词（200-300字） | 包含任务描述、功能背景、设计原则、输出约束四要素 |

### 3.4 人如何评估和保证输出质量？

AI 输出不是最终制品，必须经过人工审查。本案例的审查清单：

| 审查维度 | 检查项 | 本案例审查结果 |
|----------|--------|---------------|
| **完整性** | 备选流是否覆盖关键异常？（条件为空、日期非法、无车次、服务异常） | ✅ 覆盖了 A1-A4 四个备选流 |
| **项目一致性** | AI 输出的类名是否与项目代码一致？（如项目用 TrainController 而非 TicketQueryController） | ⚠️ 需对照 [类名映射表](类名映射表-设计名与实际项目类名对照.md) 统一 |
| **职责划分** | 边界类是否承担了业务规则？（如 TicketQueryPage 只做展示和校验，不做业务判断） | ✅ 职责划分符合设计原则 |
| **可实现性** | 备选流是否对应真实的 HTTP 状态码和错误消息？ | ⚠️ 人补充了具体的 400/401 状态码和错误消息文本 |
| **关联准确性** | 参与类的关联关系是否正确？（如 TrainStop 通过 trainId 关联 Train） | ✅ 关联关系与项目 Entity 一致 |

## 4. 工作流分析：LLM 多轮人机交互全流程

本案例是典型的 **LLM 交互式工作流**。与 Agent 的"设定目标→自主执行→人审轨迹"不同，LLM 工作流的特点是**人在每一轮都参与决策**。

### 4.1 工作流全景图

```
┌─────────────────────────────────────────────────────────┐
│  第1轮：人构建初始提示词                                   │
│  ├─ 人：梳理四要素（任务/原则/上下文/约束）                  │
│  └─ 人：写提示词，注入设计原则（边界/控制/实体类分离）        │
├─────────────────────────────────────────────────────────┤
│  第2轮：AI 生成初稿                                       │
│  ├─ AI：输出结构化设计（用例名称/参与者/事件流/参与类）       │
│  └─ AI：生成顺序图对应的文字描述                            │
├─────────────────────────────────────────────────────────┤
│  第3轮：人审阅 + 标注问题                                  │
│  ├─ 人：检查备选流是否遗漏（如"预售期外"日期校验）           │
│  ├─ 人：检查类名是否与项目一致（TrainController vs 自编名）  │
│  ├─ 人：检查职责划分（Service 不应该直接操作 HttpRequest）   │
│  └─ 人：补充项目实际文件路径和代码引用                       │
├─────────────────────────────────────────────────────────┤
│  第4轮：人调整提示词 → AI 重新生成（或人手动修改）            │
│  ├─ 策略A：小问题→人直接在输出上修改（如调整类名）            │
│  └─ 策略B：大问题→修改提示词，重新让 AI 生成                 │
├─────────────────────────────────────────────────────────┤
│  第5轮：人最终确认 + 转化为设计制品                          │
│  ├─ 人：将 AI 输出整理为正式设计文档                        │
│  ├─ 人：补充 draw.io 顺序图                                │
│  └─ 人：建立类名映射表（设计名 ↔ 项目实际类名）              │
└─────────────────────────────────────────────────────────┘
```

### 4.2 关键决策节点

| 轮次 | 决策节点 | 人做了什么 | 如果交给Agent会怎样 |
|------|----------|-----------|-------------------|
| 第1轮 | 设计原则的选择 | 人决定用"边界/控制/实体"三层分离原则 | Agent 可能默认用 MVC 分层，缺少"实体类不处理 UI"等微约束 |
| 第3轮 | 备选流粒度 | 人决定覆盖4个备选流（A1-A4），而非无限枚举 | Agent 可能列出10+备选流，或只列2个，粒度不好控制 |
| 第3轮 | 类名取舍 | 人决定保留 AI 的 `TicketQuery*` 命名体系，同时建立到项目实际类名的映射 | Agent 可能直接按项目类名生成（丢失教材的通用性），或自创类名（脱离项目） |
| 第5轮 | 输出形式 | 人将 AI 的文字描述转化为 draw.io 顺序图 | Agent 无法操作 draw.io（这是 GUI 工具） |

### 4.3 LLM 工作流的核心特征

与后续案例09（Agent 模式）对比，LLM 工作流的本质区别：

| 特征 | LLM Chat 工作流（本案例） | Agent 工作流（案例09） |
|------|--------------------------|----------------------|
| **驱动方式** | 人逐轮驱动——每轮给出新提示 | Agent 自主驱动——收到目标后自己规划步骤 |
| **人的参与密度** | 高——每轮输出都要审阅，每轮都要决策 | 低——设定目标+约束后，等 Agent 完成再审查 |
| **输出形式** | 自然语言 + 表格 + 图描述 | 可编译/可运行的代码文件 |
| **验证方式** | 人对照项目代码逐项核对 | `mvn compile` + 自动化测试 |
| **适合的任务** | 设计方案、需求分析、架构评审 | 代码生成、文件重构、批量修改 |
| **人控粒度** | 每段文字都可干预 | 只能审查最终结果和轨迹 |

## 5. 工具输入 / 提示词输入

```
### 任务描述
请为12306软件中的"车票查询"功能设计用例实现方案。

### 功能背景
用户输入出发地、目的地、出发日期后，系统查询符合条件的车次。每条车次展示：
- 车次号（如G101）
- 出发站/到达站（如北京南→上海虹桥）
- 出发时间/到达时间（如08:00-12:30）
- 历时（如4时30分）
- 各席别余票（商务座/一等座/二等座/无座等）
- 对应票价

支持按车次类型（G高速/D动车/K普速）、出发时段、是否有票筛选。

### 设计原则
1. 从用例执行流程出发设计，先梳理主事件流再补充备选流。
2. 区分边界类（负责用户交互）、控制类（协调用例流程）、实体类（表示稳定业务概念）、值对象（封装无ID的临时数据）。
3. 边界类不应承担复杂业务规则。
4. 控制类不应保存长期业务数据。
5. 实体类不应直接处理用户界面输入输出。
6. 考虑异常情况：查询条件为空、日期不合法、无符合条件车次、余票服务不可用。
7. 输出应便于后续构造设计类图。

### 输出要求
请输出：用例名称、参与者、前置条件、主事件流、备选事件流、参与类及职责、设计说明。
```

## 6. AI输出 / 工具输出示例

**用例名称**：车票查询

**参与者**：
- 主要参与者：旅客用户
- 辅助参与者：车次信息服务、余票服务、票价服务

**前置条件**：
1. 用户已进入车票查询页面（[TicketQuery.vue](../../frontend/src/views/TicketQuery.vue)）
2. 系统已维护车站数据（[stations](../../database/schema.sql) 表含~3000个全国站点）
3. 查询日期应在系统允许的预售期范围（通常15天）内

**主事件流**：
1. 用户在 `TicketQueryPage` 输入出发地（如"北京南"）、目的地（如"上海虹桥"）和出发日期
2. `TicketQueryPage` 对出发/到达站名、日期格式进行基础校验
3. `TrainController` 接收 `POST /api/trains/query` 请求，构造 `TrainQueryDTO` 查询条件
4. `TrainService.queryTrains()` 调用 `TrainMapper` 查询 `trains` 表中起始站匹配的车次
5. 对每个车次，查询 `train_stops` 获取经停站顺序
6. 查询 `ticket_inventory` 获取不同席别（`seat_types`）的余票和票价
7. `TrainController` 汇总车次、余票和票价，组装为 `TrainVO` 列表返回
8. `TicketQueryPage` 将结果渲染为车次卡片列表

**主事件流顺序图**：

> 📎 用 draw.io 打开查看：[车票查询-顺序图.drawio](车票查询-顺序图.drawio)

**顺序图说明**：

| 编号 | 步骤 | 对应事件流 |
|:---:|------|-----------|
| 1-2 | 用户输入 → 前端基础校验 | 主事件流步骤1-2 |
| **A1** | 出发地/目的地为空 | 备选流 A1 |
| **A2** | 日期不合法（早于今天） | 备选流 A2 |
| 3-5 | Controller → Service → 数据库查车次 | 主事件流步骤3-4 |
| **A3** | 无匹配车次，返回空列表 | 备选流 A3 |
| 6-7 | 逐车次查经停站和余票 | 主事件流步骤5-6 |
| **A4** | 余票服务异常时降级处理 | 备选流 A4 |
| 8-10 | 组装TrainVO → 返回前端 → 渲染卡片 | 主事件流步骤7-8 |
| **opt** | 前端筛选（车次类型/时段/是否有票） | 补充：不发起新请求 |

**备选事件流**：
- **A1-条件不完整**：出发地或目的地为空 → 前端Element Plus表单校验阻止提交，提示"请选择出发站/到达站"
- **A2-日期不合法**：日期早于今天 → 返回错误"出行日期不能为过去日期"（参考 [OrderServiceImpl](../../backend/src/main/java/com/railway/service/impl/OrderServiceImpl.java) 的日期校验逻辑）
- **A3-无车次**：指定路线无匹配车次 → 返回空列表 `data: []`，前端展示"未查询到符合条件的车次"
- **A4-余票服务异常**：`ticket_inventory` 查询失败 → 降级展示车次基础信息，标注"余票暂不可用"

**参与类及职责**（对照12306项目实际代码）：

| 类名 | 类型 | 对应项目文件 | 职责 |
|------|------|-------------|------|
| `TicketQueryPage` | 边界类 | [TicketQuery.vue](../../frontend/src/views/TicketQuery.vue) | 接收用户输入，展示查询结果，筛选控制 |
| `TrainController` | 控制类 | [TrainController.java](../../backend/src/main/java/com/railway/controller/TrainController.java) | 接收`POST /api/trains/query`，协调查询流程 |
| `TrainQueryDTO` | 值对象 | [TrainQueryDTO.java](../../backend/src/main/java/com/railway/dto/TrainQueryDTO.java) | 封装查询条件（startStation/endStation/travelDate） |
| `TrainService` | 控制类 | [TrainService.java](../../backend/src/main/java/com/railway/service/TrainService.java) | 执行车次查询业务逻辑，组装结果 |
| `Train` | 实体类 | [Train.java](../../backend/src/main/java/com/railway/entity/Train.java) | 车次基本信息（trainCode, trainType, 始发/终到站/时间） |
| `TrainStop` | 实体类 | [TrainStop.java](../../backend/src/main/java/com/railway/entity/TrainStop.java) | 车次经停站信息（站序/到站时间/发车时间） |
| `SeatType` | 实体类 | [SeatType.java](../../backend/src/main/java/com/railway/entity/SeatType.java) | 席别类型（商务座SWZ/一等座ZY/二等座ZE等10种） |
| `TicketInventory` | 实体类 | `ticket_inventory` 表 | 某车次+日期+席别的余票和票价（total/available/sold） |
| `Station` | 实体类 | [Station.java](../../backend/src/main/java/com/railway/entity/Station.java) | 车站信息（stationCode, stationName, cityName, pinyin） |
| `TrainVO` | DTO | [TrainVO.java](../../backend/src/main/java/com/railway/vo/TrainVO.java) | 汇总页面展示所需数据（车次+余票+票价） |

**设计说明**：
- `TrainController` 是流程协调者，通过 `TrainService` 整合车次查询、余票查询和票价查询
- `TrainQueryDTO` 封装前端传来的查询条件（对应 `POST /api/trains/query` 的请求体）
- `TrainStop` 和 `TicketInventory` 通过 `trainId` 与 `Train` 关联
- `TrainVO` 只用于页面结果展示，不承担业务规则判断
- 本项目实际使用 MyBatis-Plus 的 `BaseMapper` 作为数据访问层，设计图中可用 Repository 模式等价表示

## 7. 使用建议

1. **AI输出是设计初稿**：需人工审查并补充真实业务规则，如预售期天数、停运车次处理、候补票逻辑
2. **类名与项目保持一致**：如项目使用 `TrainVO` 而非 `TicketSearchResult`，应统一命名
3. **备选流要覆盖真实场景**：不只是"A/B/C"编号，要对应具体的 HTTP 状态码和错误消息
4. **设计文档作为后续案例的输入**：本案例的输出 → 案例02（类图）→ 案例09（代码生成）形成链路
5. **提示词是设计意图的载体**：写好提示词的前提是对业务和架构有清晰认识——AI不能替代人理解业务，只能加速表达
6. **区分"AI擅长的"和"人必须做的"**：AI擅长快速起草结构化文档；人必须做的是定义设计原则、判断职责边界、验证项目一致性
