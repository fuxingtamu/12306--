## 1. 案例概述

### 1.1 背景与目标

**问题场景**：项目启动时，需要将"旅客管理模块"的需求文档转化为可执行的User Story。

原始需求来自 `docs/开发计划.md` 第46-54行，内容较为简略：
- 旅客列表
- 添加旅客
- 编辑旅客
- 删除旅客
- 旅客详情
- 旅客类型

**学习目标**：
- 掌握使用AI辅助分析需求完整性的方法
- 学会将模糊需求拆解为可执行的User Story
- 记录AI辅助需求分析涉及的思想和技术

**预期成果**：
- 完整的旅客管理模块User Story清单
- 需求完整性分析报告
- 验收标准定义

## 2. 使用的AI思想和技术

### 2.1 Harness Engineering（工具工程学）

| 技术要素 | 本案例中的应用 |
|----------|----------------|
| **上下文管理** | AI读取`docs/开发计划.md`获取项目背景和技术栈上下文 |
| **输出约束** | 指导AI输出结构化Markdown格式，包含角色、行为、价值三要素 |
| **记忆系统** | 使用MEMORY.md记住用户背景（铁路售票系统）、项目特点 |
| **渐进式引导** | 分步骤引导AI：先分析完整性 → 再拆解Story → 最后补充非功能需求 |

### 2.2 Chain of Thought（思维链）

AI在分析需求时外显推理过程：

```
Step 1: 识别需求中的实体
        → 实体：旅客、旅客类型、用户

Step 2: 识别实体属性
        → 旅客属性：姓名、证件类型、证件号、旅客类型、电话...

Step 3: 识别业务操作
        → CRUD操作：查看、添加、编辑、删除

Step 4: 识别业务规则
        → 儿童需有成人的陪同关系
        → 学生、残军有优惠票资格

Step 5: 补充异常场景
        → 证件号格式校验
        → 重复旅客检测
```

### 2.3 传统软件工程的AI化

| 传统做法 | AI时代升级 | 本案例体现 |
|----------|------------|------------|
| 人工阅读需求文档 | AI快速解析 + 提取关键信息 | AI读取开发计划.md |
| 人工识别需求遗漏 | AI对比行业标准模板查漏 | AI识别6大遗漏点 |
| 手工编写User Story | AI生成结构化输出 | 一次生成15+条Story |
| 人工补充验收标准 | AI基于实体属性推导 | 每个Story自动带AC |
