#### 1.2.3 车票查询

| 编号 | 用例 | 优先级 | 前置条件 | 后置条件 |
|------|------|:---:|------|------|

| UC-T-02 | **查询车次** | P0 | 已登录 | 按出发站+到达站+日期查询，返回车次列表+随机余票 |

## 2. 核心流程时序

### 2.1 添加乘车人

![添加乘车人](diagrams/drawio/export/12a-sequence-add-passenger-new.drawio.png)

**关键设计**：
- **前端先做格式校验**：必填项、身份证格式、手机号格式在提交前校验，减少无效 API 请求
- **Service 层双校验**：① `countByUserId < 15` ② `id_type + id_number` 唯一性查询 —— 两者都做 DB 查询但分两步（先数量后唯一）
- **@TableLogic**：新增时自动设置 status=1，删除时 UPDATE 而非 DELETE
- **异常路径**：校验不通过 → `BusinessException(code, msg)` → `@RestControllerAdvice` 统一拦截 → 前端 `ElMessage.error()`

### 2.2 车票查询

![车票查询](diagrams/drawio/export/12b-sequence-query-trains-new.drawio.png)

**关键设计**：
- **车站搜索独立接口**：`GET /api/stations/search?keyword=北京`，前端 StationSelector 组件负责防抖（300ms），后端查询 `station_pinyin LIKE %keyword% OR station_name LIKE %keyword%`
- **经停匹配 SQL**：`stop_order_A < stop_order_B` 同时 JOIN 两次 `train_stops` 表（一次 for 出发站，一次 for 到达站），直达和过路车统一处理
- **余票随机生成**：遍历查询结果，对每趟车调用 `Random(0, 500)` 生成每个席别的余票数，按 `train_type` 映射席别（G→商务/一等/二等/无座，K/T/Z→软硬卧/硬座/无座）
- **筛选即时生效**：前端展开 9 大类筛选面板后重新 POST 请求（也可前端本地过滤，取决于数据量）

### 2.3 用户登录

![用户登录](diagrams/drawio/export/12c-sequence-login-new.drawio.png)

**关键设计**：
- **密码 bcrypt 单向加密**：数据库不存明文，登录时 `BCryptPasswordEncoder.matches(raw, encoded)` 比对
- **JWT 无状态**：Payload 仅含 `{sub: userId, username, iat, exp}`，每次请求 `JwtAuthFilter` 重新解析，无 Session 依赖
- **Axios 拦截器**：`utils/request.ts` 中 `request.interceptors.request.use()` 自动附加 `Authorization: Bearer <token>` 头

### 2.4 删除乘车人

![删除乘车人](diagrams/drawio/export/12d-sequence-delete-passenger-new.drawio.png)

**关键设计**：
- **软删除**：MyBatis-Plus `@TableLogic` 自动将 `DELETE` 转为 `UPDATE status=0`，所有查询自动附加 `WHERE status=1`
- **跨用户校验**：`WHERE id=? AND user_id=?` 防止用户删除他人的乘车人
- **15 人名额不扣**：软删除记录不计入 15 人上限（仅统计 `status=1` 的记录）

## 3. 领域模型与类图

### 3.1 核心类图

![类图](diagrams/drawio/export/13-class-v2-new.drawio.png)

### 3.2 实体关系说明

| 关系 | 类型 | 说明 |
|------|------|------|
| User → Passenger | 1 : 0..* | 一个用户拥有 0~15 名乘车人，通过 `user_id` 外键隔离 |
| Passenger → IDType | 引用 | 乘车人的 `idType` 字段引用 IDType 枚举表，硬编码 8 种 + DB 扩展 |
| Train → TrainStop | 1 : 1..* | 一个车次有 2~N 个经停站，通过 `stop_order` 排序 |
| TrainStop → Station | 引用 | 经停站的 `station_name` 与车站表 `station_name` 字符串匹配 |

### 3.3 类设计要点

- **Passenger 19 个字段**：last/first name 分开（适配国际证件）、ISO 3166-1 国家代码、8 种证件类型枚举、4 种旅客类型
- **软删除统一字段**：Passenger 使用 `is_deleted` 标记，其他实体统一使用 `status=0/1`
- **UML 三区表示**：每个类使用 draw.io swimlane 分为类名 / 属性 / 方法三区，分隔线区分

## 4. 乘车人状态图

### 4.1 状态机

![乘车人状态图](diagrams/drawio/export/16-passenger-state-乘车人状态图.drawio.png)

### 4.2 状态说明

| 状态 | 含义 | status 值 | 行为 |
|------|------|:---:|------|
| **NotCreated** | 未创建 | — | 乘车人尚未被添加 |
| **Active** | 正常使用中 | 1 | 可被查询、编辑、用于购票；计入 15 人上限 |
| **SoftDeleted** | 已删除（软删除） | 0 | 不计入 15 人名额，不在主列表展示；可恢复 |

### 4.3 关键转移

| 转移 | 触发 | 约束 |
|------|------|------|
| 初始 → Active | 填写信息并提交 | `count < 15` 且 `(user_id, id_type, id_number)` 唯一 |
| Active → Active | 编辑信息 | 仅可修改 `passengerType`，证件信息不可改 |
| Active → SoftDeleted | 用户确认删除 | `UPDATE status=0`（@TableLogic 自动转换） |
| SoftDeleted → Active | 恢复（手动/API） | `UPDATE status=1`，名额足够时可恢复 |
| SoftDeleted → SoftDeleted | 重复删除 | 幂等操作 |
| SoftDeleted → NotCreated | 物理删除 | ⚠ 非 MVP（管理员操作或定时清理） |
| Exists → 最终 | 账户注销 | ⚠ 非 MVP（级联删除） |

### 4.4 设计要点

- 采用**软删除**，`status` 字段区分 Active(1) 和 SoftDeleted(0)，数据不丢失
- `@TableLogic` 自动将 `DELETE` 转为 `UPDATE status=0`，所有查询自动附加 `WHERE status=1`
- **15 人上限**仅计数 `status=1` 的记录，软删除不计入
- **唯一性约束**：`(user_id, id_type, id_number)` 联合唯一索引
- 编辑仅可改 **旅客类型**（成人/儿童/学生/残军），证件类型和号码不可修改
- MVP 默认 `is_verified=1`（模拟核验通过）
