## 2. 第一阶段：AI 初稿（"能用但有坑"）

### 2.1 类图（PlantUML）

```plantuml
@startuml AI初稿
skinparam packageStyle rectangle

package "前端" {
    class TicketQueryPage <<B>> {
        + queryForm: {from, to, date}
        + submitQuery(): void
        + showResults(results: TrainVO[]): void
    }
}

package "controller" {
    class TrainController <<C>> {
        + query(@RequestBody TrainQueryDTO): Result<List<TrainVO>>
    }
}

package "service" {
    class TrainService <<C>> {
        + queryTrains(TrainQueryDTO): List<TrainVO>
    }
}

package "dto" { class TrainQueryDTO <<DTO>> { -startStation, endStation, travelDate } }
package "entity" {
    class Station <<E>> { -stationCode, stationName, cityName }
    class Train <<E>> { -trainCode, trainType, startTime, endTime }
    class TrainStop <<E>> { -stopOrder, arriveTime, departTime }
    class SeatType <<E>> { -seatTypeCode, seatTypeName, basePrice }
    class TicketInventory <<E>> { -total, available, sold, price }
}
package "vo" { class TrainVO <<VO>> { -trainCode, startStation, endStation } }

TicketQueryPage ..> TrainController
TrainController --> TrainService
TrainService ..> TrainQueryDTO
TrainService --> Train
TrainService --> TrainStop
TrainService --> TicketInventory
TrainService --> TrainVO
Train --> Station
TrainStop --> Station
TicketInventory ..> SeatType
TrainVO *-- SeatType
@enduml
```

### 2.2 初稿统计

| 维度 | 数值 |
|------|------|
| 总类数 | **10** |
| 包数 | 5（前端 / controller / service / dto / entity / vo） |
| 关系线条 | 10 条 |
| 平均每类属性数 | 3~5 个 |

### 2.3 初稿做得好的地方

- ✅ **分层结构正确**：前端 / Controller / Service / Entity / VO 五个层次清晰
- ✅ **核心链路完整**：`TicketQueryPage → TrainController → TrainService → Train/TrainStop/TicketInventory → TrainVO`
- ✅ **类型标记规范**：`<<B>>` `<<C>>` `<<E>>` `<<VO>>` `<<DTO>>` 标记到位
- ✅ **包结构合理**：按分层分包，符合 Java 项目惯例

### 2.4 初稿存在的问题

| # | 问题 | 具体表现 | 严重度 |
|---|------|---------|:---:|
| ① | **命名偏离业务** | `TrainController`/`TrainService` 语义偏"车次管理"，非"车票查询" | ⚠️ 中 |
| ② | **余票与票价混在同一实体** | `TicketInventory` 同时含 `available`/`sold` 和 `price` | 🔴 高 |
| ③ | **未定义的类型引用** | `TrainVO.seatList: List<SeatInfo>` 但 `SeatInfo` 类不在图中 | ⚠️ 中 |
| ④ | **缺少多重性标注** | 所有关联线无 `1`/`0..*`/`1..*`，无法表达业务语义 | ⚠️ 中 |
| ⑤ | **缺少 Repository 层** | Service 直接操作实体，无数据访问层抽象 | 🔵 低 |

## 3. 第二阶段：AI 精化后（"过度扩展"）

将 5 个问题反馈给 AI 进行第二轮精化后，AI 输出了以下版本。

### 3.1 类图（PlantUML，简化展示）

```plantuml
@startuml AI精化后(过度扩展)
skinparam packageStyle rectangle

package "前端" { class TicketQueryPage <<B>> }
package "controller" { class TicketQueryController <<C>> }
package "service" { class TicketQueryService <<C>> class TicketQueryValidator <<C>> }
package "dto" { class TicketQueryCondition <<DTO>> class TicketQueryResult <<DTO>> class TrainSummary <<DTO>> class StationStop <<DTO>> class SeatAvailability <<DTO>> }
package "repository" { class TrainRepository <<R>> class TrainStopRepository <<R>> class TicketInventoryRepository <<R>> class FareRepository <<R>> }
package "entity" { class Station <<E>> class Train <<E>> class TrainStop <<E>> class SeatType <<E>> class TicketInventory <<E>> class Fare <<E>> }
package "vo" { class TicketQueryResultVO <<VO>> class SeatInfoVO <<VO>> }

TicketQueryPage ..> TicketQueryController
TicketQueryController --> TicketQueryService
TicketQueryService --> TrainRepository
TicketQueryService --> TrainStopRepository
TicketQueryService --> TicketInventoryRepository
TicketQueryService --> FareRepository
TicketQueryService --> TicketQueryValidator
Train "1" --> "1..*" TrainStop
Train "1" --> "0..*" TicketInventory
Train "1" --> "0..*" Fare
@enduml
```

### 3.2 精化后统计

| 维度 | 初稿 | 精化后 | 增幅 |
|------|:---:|:---:|:---:|
| 总类数 | 10 | **26+** | +160% |
| 包数 | 5 | **7**（+repository, +多dto子包） | +40% |
| 关系线条 | 10 | **20+** | +100% |
| 新增层 | — | Repository 层、Validator | — |
| 新增中间DTO | — | `TicketQueryResult`, `TrainSummary`, `SeatAvailability`, `StationStop` | — |

### 3.3 精化后解决了什么

| 初稿问题 | AI 精化后的处理 |
|----------|---------------|
| ① 命名偏离业务 | ✅ 统一改为 `TicketQuery*` 命名体系 |
| ② 余票与票价混在一起 | ✅ 拆出独立的 `Fare` 实体类 |
| ③ 未定义的类型引用 | ✅ 补充 `SeatInfoVO` 类 |
| ④ 缺少多重性 | ✅ 关联线上标注 `1`/`1..*`/`0..*` |
| ⑤ 缺少 Repository 层 | ✅ 补充了 4 个 Repository 类 |

### 3.4 精化后引入的新问题——信息过载

AI 不是"修 5 个问题"，而是"修 5 个问题 + 补充了所有能补充的东西"：

```
一张图里同时包含：
  前端页面 + Controller + Service + Validator
  + 4 个 Repository + 5 个中间 DTO + Result 包装类
  + 6 个 Entity + 2 个 VO
  = 26+ 个类 + 20+ 条关系线
```

**这不是一张"设计类图"，而是一张"系统分层架构图 + 类图 + 数据模型图"的混合体。** 读者会被大量线条淹没，反而看不清核心的用例协作关系。

AI 的典型行为模式：

> 你问 AI "这里少了 A"，AI 不仅补上 A，还顺带补了 B、C、D、E、F——因为它认为"既然要做就做全"。但教材写作不是 API 文档，读者不需要看到所有实现细节。

## 4. 第三阶段：人工裁剪（"做减法"）

### 4.1 裁剪原则

10.4.3 小节的定位是"车票查询用例由哪些核心设计类协作完成"，不是展示全部实现细节。

| 删除 | 原因 | 移到哪里 |
|------|------|---------|
| Repository 层（4个类） | 数据访问细节 | 10.6.4 数据操作设计 |
| `TicketQueryResult`、`TrainSummary`、`SeatAvailability`、`StationStop`（4个） | 中间 DTO，增加信息量但对理解用例协作帮助不大 | 不需要 |
| `TicketQueryValidator`（1个） | 校验职责可简述为 Service 的私有方法 | 文字描述即可 |

### 4.2 裁剪后的核心类图


## 5. 三阶段逐项对比

### 5.1 命名演变

| 关注点 | 初稿 | 精化后 | 裁剪后 | 变化说明 |
|--------|------|--------|--------|---------|
| Controller | `TrainController` | `TicketQueryController` | `TicketQueryController` | 语义从"车次管理"收窄到"车票查询" |
| Service | `TrainService` | `TicketQueryService` | `TicketQueryService` | 同上 |
| 查询条件 DTO | `TrainQueryDTO` | `TicketQueryCondition` | `TicketQueryCondition` | 更准确地体现"查询条件"语义 |
| 结果 VO | `TrainVO` | `TicketQueryResultVO` | `TicketQueryResultVO` | 明确是"查询结果"而非泛化的"车次VO" |

### 5.2 实体拆分演变

| 关注点 | 初稿 | 精化后 | 裁剪后 |
|--------|------|--------|--------|
| 余票数据 | `TicketInventory`（含 `price`） | `TicketInventory`（仅余票） | `TicketInventory`（仅余票） |
| 票价数据 | （混在 `TicketInventory` 中） | 独立 `Fare` 类 | 独立 `Fare` 类 |
| 席别类型 | `SeatType` | `SeatType` | `SeatType` |
| 关系 | `TicketInventory ..> SeatType` | `TicketInventory ..> SeatType`, `Fare ..> SeatType` | 同精化后 |

> **关键教训**：AI 初稿倾向于把概念相近的数据塞进同一个类（`available` + `price` 都在 `TicketInventory`），人工审查时需要识别并拆分这些隐含的独立概念。

### 5.3 关系标注演变

| 关系对 | 初稿 | 精化后 | 裁剪后 |
|--------|------|--------|--------|
| Train → TrainStop | `-->` 无标注 | `"1" --> "1..*"` | `"1" --> "1..*"` |
| Train → TicketInventory | `-->` 无标注 | `"1" --> "0..*"` | `"1" --> "0..*"` |
| Train → Fare | （不存在） | `"1" --> "0..*"` | `"1" --> "0..*"` |
| TrainStop → Station | `-->` 无标注 | `"*" --> "1"` | `"*" --> "1"` |
| ResultVO → SeatInfoVO | `*--` 无标注 | `"1" *-- "1..*"` | `"1" *-- "1..*"` |

> 多重性标注从 0 条（初稿）→ 5 条（精化后）→ 保留 5 条（裁剪后）。这是 AI 精化贡献最大的改进点之一。

### 5.4 类数量爆炸与收敛

```
初稿:     ██████████            10 类（缺 Repository、中间DTO、Fare 等）
精化后:   ██████████████████████████  26+ 类（AI 把能加的全都加了）
裁剪后:   ████████████          12 类（删掉 Repository、中间DTO、Validator）
```

| 删除的类（精化后→裁剪后） | 数量 | 删除理由 |
|--------------------------|:---:|---------|
| Repository 层 | 4 | 数据访问细节，移至 10.6.4 |
| 中间 DTO | 4 | 增加信息量但对理解用例协作无帮助 |
| Validator | 1 | 校验逻辑应作为 Service 方法简述 |
| Result 包装类 | 1 | 框架细节，非用例设计关注点 |
| **合计删除** | **~14** | |

## 6. AI 信息过载的典型模式

对比三个阶段，可以归纳出 AI 生成设计类图时的典型行为模式：

### 6.1 AI 的三个倾向

```
倾向①：补全冲动
  "你提到了 A，那我应该把 B/C/D 也画上"
  → 导致类数量呈 2~3 倍增长

倾向②：层次下沉
  "既然画了 Service，那就应该画 Repository"
  "既然画了 Repository，那就应该画 Mapper 实现"
  → 导致设计类图变成了系统架构图

倾向③：DTO 膨胀
  "查询需要入参 DTO → 层间传递需要中间 DTO → 返回需要出参 DTO"
  → 一张图出现了 5 个 DTO（Condition, Result, Summary, Stop, Availability）
```

### 6.2 人的职责：决定"不画什么"

| AI 做的事 | 人做的事 |
|-----------|---------|
| 生成初稿结构 | ✅ 审查命名是否准确 |
| 补全属性和方法 | ✅ 审查是否多了无关属性 |
| 标注类型 `<<B>>` `<<E>>` | ✅ 审查分类是否正确 |
| 画出所有关系 | ✅ 决定哪些关系是核心的、哪些可省略 |
| 补充 Repository、Validator 等 | ✅ **决定这些类不应该出现在这一节** |

> **核心原则**：AI 负责"画什么"，人负责"不画什么"。一张好的设计类图不是把所有类都放进去，而是**恰好包含理解当前用例所需的核心类**。

## 7. 对教材写作的建议

### 7.1 教材中类图的三层控制

| 层 | 控制目标 | 本节（10.4.3）的选择 |
|----|---------|-------------------|
| 架构层 | Repository 画不画？ | ❌ 不画 → 放到 10.6.4 |
| 领域层 | Fare 要不要拆出来？ | ✅ 拆 → 余票和票价分离 |
| 展示层 | 几个 VO 够用？ | ✅ 2 个 → ResultVO + SeatInfoVO |

### 7.2 教材表述模板

案例02 第 5.5 节已有表述，这里补充一个可复用的"三段式"对比表述：

> AI 在第一轮输出中能够正确识别分层结构和核心协作链路，但存在 5 个问题：命名不准、余票与票价混杂、类型引用缺失、多重性缺省、Repository 层缺失。将这些问题反馈给 AI 进行第二轮精化后，AI 不仅修复了上述问题，还补充了 Repository 层、Validator 和多个中间 DTO，类数量从 10 个膨胀至 26 个——出现了典型的**信息过载**。经人工裁剪，移除与数据访问（Repository）、层间传递（中间DTO）、参数校验（Validator）相关的 14 个类，最终保留 12 个核心设计类。这个过程说明：**AI 善于做加法，人必须做减法**。

## 8. 关键收获

| # | 收获 | 一句话总结 |
|---|------|-----------|
| 1 | AI 初稿能命中 80% 结构 | 分层 + 核心链路基本正确，可直接用 |
| 2 | 但 20% 的细节需要人修正 | 命名、拆分、多重性、缺失引用 |
| 3 | 精化≠修复，精化=修复+扩展 | AI 会趁机把能画的都画上 |
| 4 | 人必须做减法 | 一张图 ≤15 个类是硬约束 |
| 5 | 不是删掉，是搬家 | Repository 没消失，只是移到了 10.6.4 |
| 6 | 多重性是 AI 精化最有价值的增量 | 从 0 到 5 条，且人工核实后无需修改 |
