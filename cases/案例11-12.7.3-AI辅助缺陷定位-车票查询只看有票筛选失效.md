# 案例11：AI辅助缺陷定位——"只看有票"筛选失效

> **适用章节**：12.7.3 AI辅助缺陷定位
> **工具形式**：AI日志分析 / ChatGPT / IDE Debug Assistant
> **案例功能**：车票查询
> **来源文档**：12306_AI辅助软件开发案例_10-13章_全量补齐版

---

## 1. 案例背景

车票查询中"只看有票"是常用筛选条件。用户勾选后仍看到余票为0的车次，说明筛选逻辑存在缺陷。这是一个典型的逻辑表达式错误——`||`（OR）和 `&&`（AND）混淆。

对应项目：案例09中生成的 `TicketQueryServiceImpl.filterAvailable()` 方法，实际 `TrainServiceImpl` 中也可能存在类似的布尔表达式陷阱。

---

## 2. 设计思想

AI辅助调试的关键：**提供实际行为、期望行为和相关代码片段**。与其笼统说"筛选功能坏了"，不如给出具体表达式、输入输出，让AI基于证据定位。提示词要求输出缺陷位置、原因、修复方案和测试用例。

## 3. 工具输入 / 提示词输入

```
### 任务描述
请帮助定位12306车票查询功能中"只看有票"筛选失效的问题。

### 用户反馈
用户勾选"只看有票"复选框后，页面仍然显示余票数量为0（availableCount=0）的车次。

### 相关代码（来自TicketQueryServiceImpl.filterAvailable的简化版本）
```java
return trains.stream()
        .flatMap(train -> buildResults(train, condition).stream())
        .filter(result -> condition.isOnlyAvailable()
                || result.getAvailableCount() > 0)
        .toList();
```

### 输入示例
- condition.isOnlyAvailable() = true
- result.getAvailableCount() = 0  ← 这个结果预期被过滤

### 期望行为
- 当 onlyAvailable = true 时 → 只展示 availableCount > 0 的结果
- 当 onlyAvailable = false 时 → 展示全部结果（包括 availableCount = 0 的）

### 实际行为
- onlyAvailable = true 时 → 所有结果都被保留（包括 availableCount = 0 的）

### 输出要求
1. 指出最可能的缺陷位置（精确到表达式）
2. 用真值表说明缺陷原因
3. 给出最小修复方案
4. 给出应补充的测试用例
```

## 4. AI输出 / 工具输出示例

### 1. 缺陷位置

**最可能的缺陷位置**：第3行的 filter Lambda 表达式

```java
.filter(result -> condition.isOnlyAvailable()    // ← 这里
        || result.getAvailableCount() > 0)
```

### 2. 真值表分析

| onlyAvailable | availableCount > 0 | `onlyAvailable /|/| count > 0` | 预期行为 | 是否一致 |
|:---:|:---:|:---:|:---:|:---:|
| **true** | **true (5)** | **true** ✅ 保留 | 保留 | ✅ |
| **true** | **false (0)** | **true** ❌ 保留 | **过滤** | ❌ 缺陷！ |
| **false** | true (5) | true ✅ 保留 | 保留 | ✅ |
| **false** | false (0) | true ✅ 保留 | 保留 | ✅ |

**缺陷原因**：

当前表达式 `condition.isOnlyAvailable() || result.getAvailableCount() > 0` 使用 **OR** 运算符。当用户勾选"只看有票"时，`isOnlyAvailable() = true`，由于 OR 短路求值，**整个表达式恒为 true**，导致所有结果（包括余票为0的）都被保留。

**正确的布尔逻辑**：

- 不勾选 → 保留全部：`!onlyAvailable` 为 true → 通过
- 勾选 → 仅保留有票：`onlyAvailable && count > 0` 为 true → 通过

因此正确表达式应为：`!onlyAvailable || (onlyAvailable && count > 0)`

简化后等价于：`!condition.isOnlyAvailable() || result.getAvailableCount() > 0`

### 3. 最小修复方案

```diff
 .filter(result -> 
-    condition.isOnlyAvailable() || result.getAvailableCount() > 0
+    !condition.isOnlyAvailable() || result.getAvailableCount() > 0
 )
```

**修复后的真值表验证**：

| onlyAvailable | count > 0 | `!onlyAvailable /|/| count > 0` | 结果 |
|:---:|:---:|:---:|:---:|
| true | true (5) | false /|/| true = **true** | ✅ 保留 |
| true | false (0) | false /|/| false = **false** | ✅ 过滤 |
| false | true (5) | true /|/| true = **true** | ✅ 保留 |
| false | false (0) | true /|/| false = **true** | ✅ 保留 |

### 4. 应补充的测试用例

| 用例编号 | onlyAvailable | availableCount | 预期结果 | 原因 |
|----------|:---:|:---:|:---:|------|

**JUnit测试代码骨架**：

```java
@Test
void shouldFilterNoTicketWhenOnlyAvailableIsTrue() {
    List<TrainVO> results = List.of(
        createMockVO("G101", 0),  // 无票

    );

}

@Test

    List<TrainVO> results = List.of(
        createMockVO("G101", 0),
        createMockVO("G102", 5)
    );
    List<TrainVO> filtered = filterAvailable(results, false);
    assertEquals(2, filtered.size());
}
```

## 5. 使用建议

1. **布尔表达式是最常见的小缺陷源**：`||` 和 `&&` 的混淆，取反 `!` 的位置错误，优先级问题
2. **真值表是说服力最强的调试工具**：4行真值表立刻暴露问题
3. **AI定位缺陷时输入三要素**：①实际行为 ②期望行为 ③相关代码（精确到方法/表达式）
4. **缺陷修复后必须补充测试用例**：案例12将专门讨论修复补丁的生成和验证
5. **注意null值**：若 `availableCount` 可能为 null，应增加防御性判断：`result.getAvailableCount() != null && result.getAvailableCount() > 0`
