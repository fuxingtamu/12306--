# 案例09：AI辅助代码生成——车票查询服务类代码框架

> **适用章节**：12.7.1 AI辅助代码生成
> **工具形式**：Claude Code Agent（Claude Opus 4.8，VS Code）+ 项目上下文自主感知 + MyBatis-Plus
> **工作范式**：🤖 Agent 自主执行 —— 人设定目标与约束，Agent 自主探索项目→规划步骤→生成代码→自检，人最终审查轨迹与结果
> **案例功能**：车票查询
> **来源文档**：12306_AI辅助软件开发案例_10-13章_全量补齐版

---

## 1. 案例背景

相比单行补全（案例08），车票查询服务涉及多步骤协调（校验条件→查车次→查经停→查余票→查票价→组装VO），适合使用具备项目上下文理解能力的代码智能体。

项目中对应：[TrainService.java](../../backend/src/main/java/com/railway/service/TrainService.java) 和 [TrainServiceImpl.java](../../backend/src/main/java/com/railway/service/impl/TrainServiceImpl.java)。

---

## 2. 设计思想

提示词要求智能体基于**当前项目结构**生成Service代码，明确不生成Controller、不修改数据库表结构。约束AI修改范围，降低大面积改动风险。同时要求优先复用已有Mapper/DTO。

---

## 3. 工具与方法选型

### 3.1 为什么用 Agent 而非简单 LLM + Prompt？

这是本案例最核心的选型决策。车票查询 Service 的代码生成涉及 **5个已有Mapper + 3个Entity + 2个DTO** 的跨文件协调。两种方案的对比如下：

| 对比维度 | LLM Chat + 粘贴文件（❌ 不选） | Claude Code Agent（✅ 本案例选择） |
|----------|-------------------------------|-----------------------------------|
| **项目上下文获取** | 开发者需**手动找到并粘贴** `TrainMapper.java`、`TrainVO.java`、`SeatType.java` 等8+个文件内容 | Agent **自主搜索** `com.railway.mapper` 包，读取所需文件 |
| **代码一致性** | 可能编造不存在的类名或方法（如幻觉出 `TrainRepository.findByRoute()`） | 基于**真实项目文件**生成，`TrainMapper` 确实有 `selectList()` |
| **跨文件协调** | 人需要逐个描述"TrainVO有什么字段"、"SeatType枚举值有哪些" | Agent 同时读取 Entity/VO/Mapper，自动对齐字段名和类型 |
| **约束遵守** | 靠提示词"不要生成Controller"，但 AI 可能"热心"多生成 | Agent 在项目结构中确认 Controller 已存在后，自觉跳过 |
| **前置准备** | 人工收集文件、组织上下文，约15-20分钟 | Agent 自主探索，约30秒 |
| **适用场景** | 简单函数、独立逻辑、上下文≤2个文件 | **多文件依赖、需项目感知、跨层协调**（本案例场景） |

**核心判断**：当代码生成任务满足以下**三个条件**时，Agent 优于 LLM Chat：
1. ✅ **上下文广**：需要读取 ≥3 个已有文件来保证生成代码的正确性
2. ✅ **约束多**：有明确的"不要改什么"的边界（不生成Controller、不改表结构）
3. ✅ **可自动验证**：生成后可以编译/测试来初步验证（Java 的 `mvn compile`）

本案例三条全中，因此 Agent 是最佳选择。

### 3.2 如果误用 LLM Chat 会怎样？

假设开发者用 LLM Chat 来完成这个任务，典型的问题路径：

```
开发者：请根据 TrainMapper、TrainVO、SeatType... 生成 TicketQueryServiceImpl
        [粘贴 TrainMapper.java 内容]
        [粘贴 TrainVO.java 内容]
        [粘贴 SeatType.java 内容]
        （可能遗漏 TrainStopMapper，导致生成的代码用错了Mapper）

LLM：好的，这是代码...
      （生成的代码中用了 trainRepository.findByRoute()，但项目根本不存在这个方法——幻觉）

开发者：不对，我们没有 TrainRepository，用的是 MyBatis-Plus 的 BaseMapper

LLM：抱歉，已修正...
      （但又编造了 SeatTypeMapper.selectByCode()，又一个幻觉）

开发者：算了，我自己写吧
```

Agent 模式避免了这种"人在 AI 和代码之间反复传话"的低效循环。

### 3.3 为什么不在这个案例用 Code Review Skill？

Claude Code 有内置的 `/code-review` skill，可以在代码生成后自动审查。本案例中**未使用**，原因是：
- 本案例生成的代码量约200行，人工审查可控
- 重点是展示"Agent 如何自主生成"的完整轨迹，而非审查流程
- 在实际工程中，/code-review 适合作为 Agent 生成后的**第二步**——本案例聚焦第一步

### 3.4 具体工具链

| 组件 | 本案例使用 | 说明 |
|------|-----------|------|
| **Agent 类型** | Claude Code Agent（`claude` sub-agent） | 具备文件读写、项目搜索、命令执行能力 |
| **底层模型** | Claude Opus 4.8 | 需要强代码生成能力；Sonnet 5 也可胜任但生成质量略低 |
| **IDE** | VS Code + Claude Code 扩展 | Agent 在 VS Code 终端中运行，可访问整个项目工作区 |
| **项目框架** | Spring Boot 3.2 + MyBatis-Plus 3.5 | Agent 需理解 `BaseMapper`、`LambdaQueryWrapper` 等 MyBatis-Plus 惯用模式 |
| **验证工具** | Maven（`mvn compile`） | Agent 生成后，人执行编译验证 |

### 3.5 人如何评估和保证 Agent 输出质量？

Agent 不会主动报告"我可能漏了什么"。审查是人的责任，本案例的审查框架：

| 审查层次 | 检查项 | 方法 | 本案例发现 |
|----------|--------|------|-----------|
| **L1: 编译检查** | 包名、类名、方法签名是否正确 | `mvn compile` | ✅ 编译通过 |
| **L2: API 一致性** | Agent 调用的 Mapper 方法是否真实存在？（`selectList` 是 BaseMapper 自带方法） | 核对 MyBatis-Plus 文档 | ✅ `selectList` 确实存在 |
| **L3: 业务完整性** | 5个步骤是否全部实现？（校验→查询→组装→筛选→排序） | 对照设计文档逐行检查 | ✅ 5步骤完整 |
| **L4: 性能审查** | 是否存在 N+1 查询？（`buildTrainVO` 为每个车次单独查余票） | 人工分析 Stream 调用链 | ❌ **发现 N+1 风险**（Agent 未主动识别） |
| **L5: 命名一致性** | Agent 生成的类名是否与项目一致？（`TicketQueryServiceImpl` vs 项目实际的 `TrainServiceImpl`） | 对照项目文件 | ⚠️ 需统一命名 |

**关键发现**：Agent 在 L1-L3（语法+API+业务）层面表现良好，但在 **L4 性能层面**有盲区——Agent 关注"功能正确"胜于"性能优化"。这印证了一个规律：**Agent 擅长做对，人负责做好。**

## 4. 工作流分析：Agent 自主执行全流程

本案例是典型的 **Agent 自主执行工作流**。与案例01（LLM多轮交互）不同，这里人在设定目标后"退后一步"，Agent 自主完成探索→规划→执行→自检的闭环。

### 4.1 Agent 执行轨迹详解

以下是 Claude Code Agent 在本案例中的实际工作轨迹（通过 Agent 的工具调用日志还原）：

```
┌──────────────────────────────────────────────────────────────┐
│ 阶段1: 理解任务 (Think)                                       │
│ ├─ Agent 解析提示词：需要在 service/impl/ 下生成代码            │
│ ├─ 识别约束：不生成Controller、不改表结构、复用已有Mapper       │
│ └─ 识别技术栈：Spring Boot + MyBatis-Plus + LambdaQueryWrapper │
├──────────────────────────────────────────────────────────────┤
│ 阶段2: 探索项目结构 (Explore)                                  │
│ ├─ [Glob] 搜索 com/railway/mapper/**/*.java                   │
│ │   → 发现 TrainMapper, TrainStopMapper, SeatTypeMapper       │
│ ├─ [Glob] 搜索 com/railway/entity/**/*.java                   │
│ │   → 发现 Train.java, TrainStop.java, TicketInventory.java   │
│ ├─ [Read] TrainMapper.java —— 确认有 selectList() 方法         │
│ ├─ [Read] TrainVO.java —— 确认字段：trainCode, startTime 等    │
│ ├─ [Read] TrainQueryDTO.java —— 确认字段：startStation 等      │
│ └─ [Read] TrainService.java —— 确认接口定义                    │
├──────────────────────────────────────────────────────────────┤
│ 阶段3: 规划实现 (Plan)                                         │
│ ├─ 主方法：queryTrains(TrainQueryDTO) → List<TrainVO>         │
│ ├─ 辅助方法1：validateCondition() —— 校验                      │
│ ├─ 辅助方法2：findTrainsByRoute() —— 查车次                    │
│ ├─ 辅助方法3：buildTrainVO() —— 组装单个VO                     │
│ ├─ 辅助方法4：filterAvailable() —— 只看有票筛选                │
│ └─ 辅助方法5：calculateDuration() —— 计算历时                  │
├──────────────────────────────────────────────────────────────┤
│ 阶段4: 执行生成 (Generate)                                     │
│ ├─ [Write] 生成 TicketQueryServiceImpl.java（约200行）         │
│ │   ├─ 使用构造器注入 3个 Mapper                               │
│ │   ├─ 使用 LambdaQueryWrapper 构建查询                        │
│ │   ├─ 使用 Stream API 做 map/filter/sort                     │
│ │   └─ 添加 @Service 注解和 Logger                            │
│ └─ 预留 TODO：余票查询部分标记为扩展点                          │
├──────────────────────────────────────────────────────────────┤
│ 阶段5: 自检 (Self-Check)                                       │
│ ├─ 确认使用了构造器注入（符合 Spring 最佳实践）                  │
│ ├─ 确认未生成 Controller（遵守约束）                            │
│ ├─ 确认返回类型为 List<TrainVO>（与接口一致）                   │
│ └─ 确认未修改数据库表结构（遵守约束）                            │
└──────────────────────────────────────────────────────────────┘
```

### 4.2 Agent 思考了什么（关键推理节点）

通过分析 Agent 的工具调用序列，可以还原其推理过程：

| 步骤 | Agent 的行为 | 背后的推理 |
|------|-------------|-----------|
| 先 Glob 搜索 Mapper | 在生成代码前先搞清楚有哪些 Mapper 可用 | "我不能编造 Mapper 名称，必须先看看项目里有什么" |
| 逐个 Read Mapper/Entity/VO | 确认每个类的方法签名和字段名 | "`TrainVO` 有没有 `setSeatList()`？`TrainMapper` 有没有 `selectList()`？必须确认" |
| 选择 LambdaQueryWrapper | 在提示词中提到优先使用 | "提示词明确说了 MyBatis-Plus LambdaQueryWrapper，我的代码要体现这个技术选型" |
| 预留 TODO 而非编造 | 余票查询部分标记 `// TODO` | "`ticket_inventory` 表对应哪个 Mapper 我没找到，不能编造，标记 TODO 让人补充" |

**可观测性**：Agent 的每一步操作都留下了工具调用记录（Read 了什么文件、Write 了什么内容），这使得人可以**回溯审查**——如果生成的代码有问题，可以定位到"Agent 读了什么"和"Agent 没读什么"。

### 4.3 Agent 工作流的核心特征

与案例01（LLM 交互式）对比：

| 特征 | Agent 自主执行（本案例） | LLM 多轮交互（案例01） |
|------|------------------------|----------------------|
| **驱动方式** | Agent 自主驱动——收到目标后自己规划步骤 | 人逐轮驱动——每轮给出新提示 |
| **人的参与密度** | 低——设定目标+约束后，等 Agent 完成再审查 | 高——每轮输出都要审阅，每轮都要决策 |
| **人的参与节点** | **两个节点**：①设定目标时（写提示词+约束）②审查结果时（L1-L5检查） | **每轮都参与**：审输出→调提示词→再看输出 |
| **输出形式** | 可编译的 `.java` 文件 | 自然语言 + 表格 + 图描述 |
| **验证方式** | `mvn compile` + 人工 L1-L5 审查 | 人对照项目代码逐项核对 |
| **适合的任务** | 代码生成、文件重构、批量修改——规则明确、可自动验证 | 设计方案、需求分析、架构评审——需要人的判断力 |
| **Agent 的价值** | 省去"人工探索项目结构+收集上下文"的时间 | — |
| **人的价值** | 发现 Agent 盲区（N+1查询、命名一致性、性能风险） | 做设计决策、定义原则、判断取舍 |
| **失败代价** | 中——生成的代码编译不通过，修改或放弃 | 低——设计文档可以手动修改 |
| **重试策略** | 修改提示词约束 → 让 Agent 重新生成 → 对比两个版本 | 人在同一轮对话中追问 → AI 修正输出 |

### 4.4 如何检验 Agent 的正确性？

```
检验流程（由人执行）:

1. mvn compile
   ├─ 通过 → 进入第2步
   └─ 失败 → 检查编译错误，判断是 Agent 错误还是依赖缺失

2. 对照设计文档检查步骤完整性
   ├─ validateCondition() 是否校验了出发站/到达站/日期？
   ├─ findTrainsByRoute() 是否用了正确的 Mapper？
   ├─ buildTrainVO() 是否组装了所有必要字段？
   ├─ filterAvailable() 的布尔逻辑是否正确？（这是案例11/12的伏笔）
   └─ 排序是否按出发时间升序？

3. 检查代码是否遵守约束
   ├─ 有没有多生成 Controller？→ 没有 ✅
   ├─ 有没有修改数据库表结构？→ 没有 ✅
   ├─ 有没有复用已有 Mapper？→ 复用了 TrainMapper 等 ✅

4. 识别 Agent 盲区
   ├─ N+1 查询？→ buildTrainVO 内逐车次查询余票 ❌
   ├─ 异常处理？→ 只有 IllegalArgumentException，缺少自定义业务异常
   ├─ 日志记录？→ 有基础的 log.info，但缺少关键步骤的日志
   └─ null 安全？→ filterAvailable 未防御 availableCount=null ❌
```

## 5. 工具输入 / 提示词输入

```
请基于当前项目结构（com.railway），在 service/impl/ 下生成或完善 TicketQueryServiceImpl。

### 业务规则
1. 输入：TrainQueryDTO（startStation, endStation, travelDate, trainType, onlyAvailable）
2. 校验：出发站/到达站/日期必填，日期不可为过去
3. 查询：通过TrainMapper查询trains表，通过TrainStopMapper查询经停站
4. 余票：通过ticket_inventory表查询指定日期+车次的各席别余票和票价
5. 组装：将Train + TrainStop + TicketInventory 组装为 TrainVO 列表
6. 筛选：如 onlyAvailable=true，过滤掉所有席别available=0的车次
7. 排序：按出发时间升序返回

### 约束
- 不要生成Controller，Controller已有
- 不要修改数据库表结构
- 优先复用已有 Mapper（TrainMapper, TrainStopMapper, SeatTypeMapper）
- 使用 MyBatis-Plus LambdaQueryWrapper 构建查询
- 返回类型为 List<TrainVO>

### 输出
生成完整的 TicketQueryServiceImpl.java，包含：
- 主方法 queryTrains
- 私有辅助方法（校验、组装、筛选）
```

**提示词设计要点**（与案例01的对比）：

| 提示词要素 | 案例01（LLM设计） | 本案例（Agent代码生成） |
|-----------|-------------------|------------------------|
| 任务描述 | 开放式的"设计方案" | 具体的"在 service/impl/ 下生成文件" |
| 业务规则 | 功能背景描述 | **精确的**输入/输出/步骤定义 |
| 设计原则 | 抽象原则（边界/控制/实体分离） | **具体的**技术约束（用 LambdaQueryWrapper，不要生成 Controller） |
| 输出约束 | 内容结构约束（用例名称→事件流→参与类） | **文件级**约束（完整的 .java 文件，含主方法和辅助方法） |

规律：**Agent 的提示词需要比 LLM Chat 更精确的技术约束**——因为 Agent 会自主执行，人没有中间纠偏的机会。

## 6. AI输出 / 工具输出示例

```java
package com.railway.service.impl;

import com.baomidou.mybatisplus.core.conditions.query.LambdaQueryWrapper;
import com.railway.dto.TrainQueryDTO;
import com.railway.entity.*;
import com.railway.mapper.*;
import com.railway.service.TrainService;
import com.railway.vo.TrainVO;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.springframework.stereotype.Service;

import java.time.LocalDate;
import java.util.*;
import java.util.stream.Collectors;

@Service
public class TicketQueryServiceImpl implements TrainService {

    private static final Logger log = LoggerFactory.getLogger(TicketQueryServiceImpl.class);

    private final TrainMapper trainMapper;
    private final TrainStopMapper trainStopMapper;
    private final SeatTypeMapper seatTypeMapper;

    public TicketQueryServiceImpl(TrainMapper trainMapper,
                                   TrainStopMapper trainStopMapper,
                                   SeatTypeMapper seatTypeMapper) {
        this.trainMapper = trainMapper;
        this.trainStopMapper = trainStopMapper;
        this.seatTypeMapper = seatTypeMapper;
    }

    /**
     * 车票查询主方法
     */
    public List<TrainVO> queryTrains(TrainQueryDTO condition) {
        // Step 1: 校验条件
        validateCondition(condition);

        // Step 2: 查询匹配车次
        List<Train> trains = findTrainsByRoute(condition);

        // Step 3: 为每个车次组装详细信息（余票+票价）
        List<TrainVO> results = trains.stream()
                .map(train -> buildTrainVO(train, condition.getTravelDate()))
                .collect(Collectors.toList());

        // Step 4: "只看有票"筛选
        if (Boolean.TRUE.equals(condition.getOnlyAvailable())) {
            results = filterAvailable(results);
        }

        // Step 5: 按出发时间排序
        results.sort(Comparator.comparing(TrainVO::getStartTime));

        log.info("车票查询结果: 路线={}→{}, 日期={}, 结果数={}",
                condition.getStartStation(), condition.getEndStation(),
                condition.getTravelDate(), results.size());

        return results;
    }

    /**
     * 校验查询条件
     */
    private void validateCondition(TrainQueryDTO condition) {
        if (condition == null) {
            throw new IllegalArgumentException("查询条件不能为空");
        }
        if (condition.getStartStation() == null
                || condition.getStartStation().isBlank()) {
            throw new IllegalArgumentException("出发站不能为空");
        }
        if (condition.getEndStation() == null
                || condition.getEndStation().isBlank()) {
            throw new IllegalArgumentException("到达站不能为空");
        }
        if (condition.getTravelDate() == null) {
            throw new IllegalArgumentException("出发日期不能为空");
        }
        // 日期不可为过去
        LocalDate travelDate = LocalDate.parse(condition.getTravelDate());
        if (travelDate.isBefore(LocalDate.now())) {
            throw new IllegalArgumentException("出行日期不能为过去日期");
        }
    }

    /**
     * 根据出发/到达站查询车次
     */
    private List<Train> findTrainsByRoute(TrainQueryDTO condition) {
        // 通过经停站表查找：出发站和到达站都在经停序列中，且出发站在到达站之前
        // 简化实现：匹配start_station_name和end_station_name
        LambdaQueryWrapper<Train> wrapper = new LambdaQueryWrapper<>();
        wrapper.eq(Train::getStartStationName, condition.getStartStation())
               .eq(Train::getEndStationName, condition.getEndStation());
        if (condition.getTrainType() != null) {
            wrapper.eq(Train::getTrainType, condition.getTrainType());
        }
        return trainMapper.selectList(wrapper);
    }

    /**
     * 组装单条车次的展示VO（含各席别余票和票价）
     */
    private TrainVO buildTrainVO(Train train, String travelDate) {
        TrainVO vo = new TrainVO();
        vo.setTrainId(train.getId());
        vo.setTrainCode(train.getTrainCode());
        vo.setTrainType(train.getTrainType());
        vo.setStartStation(train.getStartStationName());
        vo.setEndStation(train.getEndStationName());
        vo.setStartTime(train.getStartTime().toString());
        vo.setEndTime(train.getEndTime().toString());
        // 计算历时
        vo.setDuration(calculateDuration(train));

        // TODO: 查询ticket_inventory表获取实时余票和票价
        // 当前为简化实现，预留扩展点
        vo.setSeatList(new ArrayList<>());

        return vo;
    }

    /**
     * "只看有票"筛选：过滤掉所有席别余票为0的车次
     */
    private List<TrainVO> filterAvailable(List<TrainVO> results) {
        return results.stream()
                .filter(vo -> vo.getSeatList().stream()
                        .anyMatch(seat -> seat.getAvailable() != null
                                && seat.getAvailable() > 0))
                .collect(Collectors.toList());
    }

    /**
     * 计算车次历时（格式：X时X分）
     */
    private String calculateDuration(Train train) {
        long minutes = java.time.Duration.between(
                train.getStartTime(), train.getEndTime()).toMinutes();
        return (minutes / 60) + "时" + (minutes % 60) + "分";
    }
}
```

## 7. 使用建议

1. **代码智能体生成后先编译**：立刻 `mvn compile` 检查类路径、包名、方法签名
2. **核对Mapper方法**：`TrainMapper` 是否有 `selectList(wrapper)`？这是 MyBatis-Plus 的 `BaseMapper` 自带方法，确认即可
3. **N+1查询风险**：`buildTrainVO` 为每个车次单独查询余票——若一次查询返回20个车次，会产生20次余票查询。生产环境建议改为一条JOIN SQL或批量查询
4. **智能体更懂项目结构**：Claude Code/Cursor 能读取 `@MapperScan("com.railway.mapper")` 和已有Entity，生成代码更准确
5. **Agent 的输出是"起点"不是"终点"**：Agent 擅长快速生成框架代码（80%），剩下的边界处理、性能优化、异常处理（20%）需要人补充
6. **Agent 提示词要"紧"不要"松"**：约束写得越具体（"使用 LambdaQueryWrapper"、"不要生成 Controller"），Agent 的输出越可控。松散的提示词等于把决策权完全交给 Agent
7. **建立 Agent 输出的审查清单**：L1编译→L2 API→L3业务→L4性能→L5规范，形成团队的 Agent 代码审查 SOP
