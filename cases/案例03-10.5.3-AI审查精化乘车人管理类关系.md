# 案例03：AI审查并精化"乘车人管理"类关系

> **适用章节**：10.5.3 精化类间的关系
> **工具形式**：AI设计审查助手 + 现有类设计草稿
> **案例功能**：乘车人管理
> **来源文档**：12306_AI辅助软件开发案例_10-13章_全量补齐版

---

## 1. 案例背景

乘车人管理功能覆盖添加、修改、删除、查询乘车人，涉及身份证件/手机号等敏感信息，以及与订单功能的关联。当前12306项目中相关类包括：[PassengerController](../../backend/src/main/java/com/railway/controller/PassengerController.java)、[PassengerService](../../backend/src/main/java/com/railway/service/PassengerService.java)、[Passenger](../../backend/src/main/java/com/railway/entity/Passenger.java)。

---

## 2. 设计思想

不让AI从零生成类关系，而是把已有类设计交给AI审查，更接近真实开发的设计评审。从职责划分、耦合关系、隐私风险、异常流程角度提出精化建议。

## 3. 工具输入 / 提示词输入

```
### 任务描述
请审查12306软件"乘车人管理"功能的类间关系，并提出精化建议。

### 当前类设计（对应12306项目实际代码）
PassengerController — 负责添加、修改、删除、查询、设为默认乘车人（6个API端点）
PassengerService — 应用服务，调用Mapper完成CRUD
Passenger — 实体，字段见下方
PassengerMapper — MyBatis-Plus BaseMapper
前端有4个版本的Passenger页面（Normal/Designer/Mobile/基础版）

Passenger当前字段：
- id, userId, name, idType(证件类型), idCard(证件号)
- passengerType(乘客类型: 成人/儿童/学生/残军)
- phone, nationality, birthDate, gender, expireDate, email
- isDefault, isVerified, createTime, updateTime

### 业务约束（来自实际需求）
1. 同一用户不能重复添加相同证件号码的乘车人
2. 删除乘车人时，如存在未完成订单（待支付/已支付未出行），不允许删除
3. 证件号码和手机号展示时必须脱敏（前端Passengers-Mobile.vue已实现）
4. 控制类不应直接处理复杂校验规则
5. 每个用户最多30名乘车人

### 输出要求
请输出：当前关系的问题、推荐类间关系（标注依赖方向）、修改后的关系说明
```

## 4. AI输出 / 工具输出示例

**当前设计的问题**：

| 问题 | 位置 | 影响 |
|------|------|------|
| ① PassengerController 直接持有 PassengerService 引用，无中间层 | [PassengerController.java](../../backend/src/main/java/com/railway/controller/PassengerController.java) | Controller知晓了过多业务细节 |
| ② 证件号脱敏逻辑分散在前端4个Passenger页面中分别实现 | Passengers-Mobile.vue / Passengers-Normal.vue 等 | 脱敏规则不一致风险 |
| ③ 删除约束检查（未完成订单）应在Service层但容易被Controller绕过 | PassengerController.delete() | 可能产生数据一致性问题 |
| ④ 无专门的身份核验状态管理类 | Passenger.isVerified 仅Boolean | 无法区分"待核验/通过/失败"等状态 |
| ⑤ 30人上限校验未在Service层统一拦截 | 添加接口 | 前端校验可被绕过 |

**推荐类间关系**（`-->` = 依赖/调用，`..>` = 数据传递）：

```
PassengerController
    --> PassengerApplicationService (新建，协调用例流程)
        --> PassengerValidator (新建，集中校验：重复证件号、30人上限)
        --> PassengerPrivacyService (新建，脱敏：证件号/手机号)
        --> PassengerUsageChecker (新建，检查未完成订单)
        --> PassengerMapper (已有)
        --> Passenger (已有，纯实体)
    --> PassengerDTO (新建，用于返回脱敏后的数据)
```

**修改后关系说明**：

| 组件 | 职责 | 与原设计对比 |
|------|------|-------------|
| `PassengerApplicationService` | 核心协调类，组织"校验→保存→脱敏→返回"流程 | 原`PassengerService`职责过于宽泛，拆分后更聚焦 |
| `PassengerValidator` | 集中执行"同一用户同一证件号不重复"、"最多30名"等规则 | 新增：原逻辑散落在前端+Service中 |
| `PassengerPrivacyService` | 证件号/手机号脱敏规则统一实现 | 新增：原逻辑分布在4个前端页面中 |
| `PassengerUsageChecker` | 删除前检查该乘车人是否有关联的未完成订单 | 新增：防止绕过约束直接删除 |
| `PassengerMapper` | 保持 MyBatis-Plus BaseMapper，仅负责数据持久化 | 不变 |
| `PassengerDTO` | 返回给前端的脱敏后数据（maskedIdCard, maskedPhone） | 新增：避免实体直接暴露完整证件号 |

**调用链示例 — 添加乘车人**：

```
POST /api/passengers
    → PassengerController.add()
    → PassengerApplicationService.addPassenger(userId, dto)
        → PassengerValidator.checkDuplicate(idCard)    // 重复检查
        → PassengerValidator.checkLimit(userId)         // 30人上限
        → PassengerMapper.insert(passenger)             // 保存
        → PassengerPrivacyService.mask(passenger)        // 脱敏
        → 返回 PassengerDTO
```

## 5. 使用建议

1. **AI建议的新增类要控制数量**：本案例建议新增4个类，实际可根据项目大小合并（如Validator和Checker可合并为PassengerGuardService）
2. **脱敏是独立关注点**：`PassengerPrivacyService` 应独立出来，不同证件类型有不同的脱敏规则（身份证保留前4后4，护照保留首尾）
3. **检查现有代码后决定**：该项目的 [PassengerServiceImpl](../../backend/src/main/java/com/railway/service/impl/PassengerServiceImpl.java) 已实现部分校验，审查前应先通读
4. **类图要反映真实依赖方向**：Controller → Service → Mapper 是单向的，不允许反向依赖
