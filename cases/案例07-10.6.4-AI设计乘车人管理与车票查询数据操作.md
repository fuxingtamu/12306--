## 1. 案例背景

乘车人管理涉及新增/修改/删除/查询/脱敏，车票查询涉及站点/车次/经停/余票/票价多表关联。两类功能都需明确数据操作顺序、事务边界和隐私保护。对应项目数据库：[schema.sql](../../database/schema.sql)（14张表）。

---

## 2. 设计思想

数据操作设计不只是写表结构，还要说明何时查、何时插、何时更新，哪些操作需要事务。提示词用"伪SQL"避免AI编造过于具体但无法运行的SQL。重点覆盖隐私保护和业务约束。

## 3. 工具输入 / 提示词输入

```
### 任务描述
请为12306软件中的"乘车人管理"和"车票查询"功能设计数据操作。

### 功能范围
1. 乘车人管理：添加/修改/删除/查询乘车人列表
2. 车票查询：根据出发站、到达站、日期查询车次、经停、余票和票价

### 数据表（来自项目schema.sql）
passengers — 乘车人：id, user_id, name, id_type, id_card, passenger_type, phone,

trains — 车次：id, train_code, train_type, start_station_name, end_station_name,
         start_time, end_time, total_time, total_stops
train_stops — 经停站：id, train_id, station_id, stop_order, arrive_time, depart_time

ticket_inventory — 库存：id, train_id, travel_date, seat_type, total, available, sold, price
seat_types — 席别：id, seat_type_code, seat_type_name, base_price
orders — 订单：id, user_id, order_no, order_status(0待支付/1已支付/2已取消...)

1. 同一用户不能重复添加相同证件号的乘车人
2. 存在未完成订单时不允许删除乘车人
3. 查询乘车人列表时不返回完整证件号码（脱敏）
4. 车票查询支持"只看有票"过滤
5. 每个用户最多30名乘车人

### 输出要求
```

### 二、关键伪SQL

**OP-1：添加乘车人（含重复检查）**

```sql
-- Step 1: 检查重复证件号 + 用户上限
SELECT COUNT(*) AS cnt
FROM passengers
WHERE user_id = :userId
  AND id_card = :idCard
  AND status != 1; -- 未逻辑删除

-- 如 cnt > 0 → 返回错误"该证件号已被添加"

SELECT COUNT(*) AS total
FROM passengers
WHERE user_id = :userId
  AND status != 1;

-- 如 total >= 30 → 返回错误"最多添加30名乘车人"

-- Step 2: 插入
INSERT INTO passengers (
    user_id, name, id_type, id_card, passenger_type,
    phone, nationality, birth_date, gender, expire_date, email,
    is_default, is_verified, create_time, update_time
) VALUES (
    :userId, :name, :idType, :idCard, :passengerType,
    :phone, :nationality, :birthDate, :gender, :expireDate, :email,
    0, 0, NOW(), NOW()
);
```

**OP-3：删除乘车人（含未完成订单检查）**

```sql
-- Step 1: 检查未完成订单（待支付/已支付未出行）
SELECT COUNT(*) AS pending_cnt
FROM order_items oi
JOIN orders o ON oi.order_id = o.id
WHERE oi.passenger_id = :passengerId
  AND o.order_status IN (0, 1); -- 0待支付 1已支付(未出行)

-- 如 pending_cnt > 0 → 返回错误"该乘车人有未完成订单，暂不可删除"

-- Step 2: 逻辑删除
UPDATE passengers
SET status = 1, update_time = NOW()
WHERE id = :passengerId AND user_id = :userId;
```

**OP-5：车票查询（多表关联）**

```sql
-- 查询某日某路线车次 + 余票 + 票价
SELECT
    t.id, t.train_code, t.train_type,
    t.start_station_name, t.end_station_name,
    t.start_time, t.end_time, t.total_time,
    st.seat_type_code, st.seat_type_name,
    ti.available, ti.price
FROM trains t
JOIN train_stops ts1 ON t.id = ts1.train_id
JOIN train_stops ts2 ON t.id = ts2.train_id AND ts2.stop_order > ts1.stop_order
LEFT JOIN ticket_inventory ti ON t.id = ti.train_id
    AND ti.travel_date = :travelDate
LEFT JOIN seat_types st ON ti.seat_type_code = st.seat_type_code
WHERE ts1.station_name = :startStation
  AND ts2.station_name = :endStation
  AND (:trainType IS NULL OR t.train_type = :trainType)
ORDER BY t.start_time ASC;
```

### 三、事务控制建议

| 操作 | 事务策略 | 原因 |
|------|----------|------|
| 添加乘车人 | `@Transactional` — 重复检查+INSERT | 检查与插入之间不应有并发插入 |
| 删除乘车人 | `@Transactional` — 订单检查+逻辑删除 | 检查与删除之间不应有状态变化 |
| 修改乘车人 | 单条UPDATE，不需显式事务 | 乐观锁：`WHERE id=? AND update_time=?` 防并发覆盖 |
| 车票查询 | 只读事务或无需事务 | 多表关联查询，设置 `readOnly=true` 可优化 |
| 购票（关联操作） | `@Transactional` — 库存扣减+订单创建 | 库存乐观锁（version字段）防超卖 |

### 四、数据安全建议

| 场景 | 措施 |
|------|------|
| 证件号存储 | 数据库可考虑 AES-256 加密存储，或至少保证传输层 TLS |
| 查询列表返回 | 接口层（Controller→前端）脱敏：身份证 `1101****5678`，手机 `138****8000` |
| 日志打印 | 禁止在日志中输出完整 `idCard`/`phone`；使用 `masked` 方法处理 |
| SQL注入防护 | MyBatis-Plus 参数化查询（`#{param}`）默认防注入，避免 `${param}` |
| 敏感字段索引 | `id_card` 建立索引用于重复检查，但需评估索引泄露风险 |

---

## 5. 使用建议

1. **伪SQL要能演化成真实SQL**：表名/字段名/索引名应与项目 schema.sql 一致
2. **车票查询是高频操作**：建议加 Redis 缓存热门路线（北京→上海、广州→深圳等）
3. **事务粒度**：只包裹必须一致的操作（检查+写入），不要把只读查询也包进长事务
4. **敏感字段脱敏分层**：DB加密 → 实体类保存原始 → Service脱敏 → DTO返回脱敏后数据
